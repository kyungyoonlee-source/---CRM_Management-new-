현재 데이터 구조 및 한계
⚠️ 현재 방식 (localStorage)
• 데이터가 각자 브라우저에만 저장
• 팀원 간 실시간 공유 불가
• PC 바꾸면 데이터 초기화
• 동시 접속 시 충돌 위험
✅ 목표 방식 (공유 DB)
• 모든 팀원이 같은 데이터 실시간 공유
• 어느 PC/폰에서든 동일한 데이터
• 데이터 안전 보관 (클라우드)
• 팀별 접근 권한 분리 가능
🗺️ 데이터 공유 전환 로드맵
STEP 1
즉시
JSON 백업 공유 방식
현재 지원 중. 팀장 PC에서 데이터 입력 → JSON 백업 → 슬랙/이메일로 공유 → 팀원 복원. 완벽하지 않지만 당장 사용 가능한 방법.
STEP 2
1~2주
Google Sheets 연동
현재 Google Sheets로 관리 중인 기관 정보를 이 시스템에서 불러오고 저장. 팀원 모두가 같은 Sheets에 접근 가능. 개발자 없이도 구현 가능.
STEP 3
1~2개월
Supabase (무료 클라우드 DB) 연동
팀원 모두 실시간 공유, 팀별 로그인 권한 분리, 데이터 영구 보관. 무료 플랜으로 시작 가능. 개발자 1명이 반나절이면 셋업 가능.
STEP 4
장기
자체 앱 VOC 자동 연동
자체 앱에서 VOC 발생 시 이 시스템에 자동 등록. OPS팀이 별도 입력 없이 실시간 확인 가능. API Webhook 방식으로 구현.

연구원별 월간 활동 추이
현장 컨설팅 가이드
목적
  • 고객의 이해도 및 만족도를 향상시켜 스토리라인 서비스 안정적 안착, 유지 및 재계약 유도
  • 현장 VoC 내부 환류(사례공유회) 및 전사 방향성 재정렬
연구원 역할
  • 현장의 실태 조사 및 파악
  • 실태 취합 및 내부 공유를 통해 자사 내 팀별 업무 방향성 정렬(영업 포함)에 기여
  • 현장 니즈에 맞춘 즉각적 대응 및 후속 관리
  • 현장 니즈에 맞춘 즉각적 대응 외 연구(기관 맞춤별 지원 방법, 교육 기획 및 운영 등)
  • 사례 취합
방법
단순 원리 적용 : 문제 진단 → 기관 맞춤형 지원 → 기록 및 기관별 온도 체크 및 분리 → 온도에 따른 기관별 방문 시기 계획 후 재방문 (이 과정을 반복하며 기관별 활용 역량(활용 독립) 향상을 위해 기여)
이유
  1. 기관별 상황(사용 연령, 교사 및 부모 특성, 유아 개인차)의 차이가 크므로 필요로 하는 지원이 각각 다름
  2. 교직원들은 기관 상황에 맞게 스토리라인을 활용 및 커스터마이징 할 수 있는 가장 좋은 리소스를 가지고 있는 인적 자원임.
  3. 현 상황 기준 자사에 가장 필요한 정보는 기관별 실태 파악임. 이 후, 업무 우선순위 및 구체적 컨설팅 계획 수립이 가능함.
상반기 진행 계획
1) 현장 인터뷰(실태 파악) 진행 (p3-4 참고)
  • 목적 : 1차 문제 진단을 위해, 그 첫 시작을 현장 인터뷰(실태 파악)지 항목을 기반으로 조사
  • 시기 : 2026년 4월부터 지역별 담당 연구원이 해당 기관에 어떠한 이유로든 처음 방문하는 시점(교육, 간담회, 라포 형성을 위한 방문 등 모두 포함)
  • 사후 검증 계획 : 1년 후 재조사를 통해 자사의 작용(교육, 컨설팅, R&D 등) 효과를 검증
2) 데이터 취합 및 월별 공유회 진행 (현수님 컨펌 완)
  • 연구원별 취합 및 데이터 공유 방법 (논의 및 시스템 개발 중)

📁 현장 파일 공유 규칙
원칙: 현장에서 수집된 모든 자료(사진, 메모, 체크리스트)는 방문 당일 CRM에 등록한다.

방법: 방문 기록 탭 → '컨설팅' 또는 '교육' 유형으로 등록 → 내용에 핵심 포인트 요약

금지사항: 개인 메신저에만 공유하고 CRM 미등록하는 행위. 중복 등록 방지를 위해 등록 전 기존 기록 확인 필수.

📌 주간 팀 미팅 전날까지 해당 주 방문 기록 전건 CRM 입력 완료
2
📊 데이터 기반 대화 원칙
원칙: 기관과의 모든 컨설팅 대화는 CRM 데이터(VOC 건수, 교육 횟수, 등급 변동)를 기반으로 한다.

활용법: 방문 전 대시보드에서 해당 기관 상세 확인 → 현장에서 데이터 근거 제시 → 개선 방향 합의

효과: 주관적 판단 최소화, 기관 신뢰 확보, 일관된 메시지 전달

📌 "지난 분기 대비 VOC 2건 감소, 교육 완료율 향상" 같은 구체적 수치 활용 권장
3
🏁 VOC 종결권 기준
원칙: VOC 상태를 '처리완료'로 변경하는 권한은 최종 확인자(OPS팀장 또는 담당 연구원 팀장)에게 있다.

자체 앱 VOC 연동
연동 예정
어떻게 작동하나요?
자체 앱에서 사용자(기관)가 VOC를 남기면 → 자동으로 이 CRM에 VOC가 등록됩니다.
OPS팀이 앱을 따로 확인하지 않아도 CRM 하나에서 모든 VOC를 볼 수 있습니다.
구현을 위해 개발팀에 전달할 내용
방식: Webhook (HTTP POST 요청)
조건: 앱에서 VOC 등록 시 자동 트리거
전달 데이터: 기관ID, 기관명, VOC내용, 등록일시, 유형
수신 주소: CRM API 엔드포인트 (Supabase 또는 자체 서버)
인증: API Key 방식
💡 지금 당장 할 수 있는 것: 앱 VOC를 CSV로 정기 내보내기 → 이 시스템 CSV 업로드로 수동 동기화 (주 1회 권장)

Google Sheets 연동
연동 예정
어떻게 작동하나요?
현재 Google Sheets에 있는 전국 기관 정보를 이 CRM에서 바로 불러오거나 저장합니다.
Sheets 수정 → CRM 반영, CRM 수정 → Sheets 반영이 자동으로 이루어집니다.
지금 당장 할 수 있는 방법 (3단계)
1
Google Sheets에서 기관 목록을 CSV로 내보내기 (파일 → 다운로드 → CSV)
2
CRM '기관 관리' 탭 → 'CSV 업로드' 버튼으로 가져오기
형식: 기관명, 담당연구원, 지사, 지역, 원장명, 연락처, 연차, 등급, 특징
3
CRM에서 업데이트한 내용은 'CSV 내보내기'로 다운로드 → Sheets에 붙여넣기
🚀 자동 실시간 연동을 원하신다면: Google Sheets API + Apps Script 연동이 필요합니다.
Google Cloud Console에서 Sheets API 활성화 → API Key 발급 → 이 시스템에 연결.
제공 필요 정보: Google Sheets URL(공유 링크), Google 계정
📋 연동 우선순위 권장 순서
1순위 (즉시) — CSV 수동 동기화 (Google Sheets ↔ CRM)
2순위 (1~2주) — Google Sheets API 자동 연동
3순위 (1~2개월) — Supabase 공유 DB 전환 (전팀 실시간 공유)
4순위 (장기) — 자체 앱 Webhook VOC 자동 등록
